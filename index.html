Abstract
This report documents the design and implementation of a Classroom/Lab Access Control System developed for EFB 2073 - Microprocessors and Computer Architecture using the NUCLEO-F401RE Cortex-M4 microcontroller and the ARM Mbed development environment. The system integrates a 4x4 matrix keypad for PIN-based authentication, a DHT11 digital sensor for real-time temperature and humidity monitoring, an SSD1306 OLED display for system feedback, and three LED indicators for visual status signalling. Access is granted only when both conditions are satisfied: the correct 4-digit PIN is entered and the room temperature is below 35 C. Overheating triggers an automatic access denial with a distinct yellow LED warning regardless of PIN correctness. Additional features include a 30-second lockout after three consecutive failed attempts, an admin mode for PIN reconfiguration via the onboard push button, occupancy tracking with live status display, and a rolling access log storing the last three entry attempts. All firmware was written in C++ using the Mbed OS framework, with no external libraries required. All features were successfully verified through hardware testing on the NUCLEO board.
Section 1

Background

1.1 Project Introduction

Physical access control is a fundamental requirement in laboratory and research environments where sensitive equipment, intellectual property, and safety-critical conditions must be protected. Traditional key-based systems offer no audit trail and cannot respond dynamically to environmental conditions such as overheating. Microcontroller-based electronic access control systems address these limitations by combining credential-based authentication with real-time environmental sensing.

This project implements a Classroom/Lab Access Control System on the NUCLEO-F401RE development board, equipped with the STM32F401RE microprocessor based on the ARM Cortex-M4 32-bit architecture. The ARM Mbed OS framework is used as the software development environment, providing a hardware abstraction layer and a rich set of peripheral driver APIs suited to rapid embedded system prototyping. The system builds directly on the firmware development skills acquired in Practical Tasks #1 and #2, extending them into a complete multi-peripheral integrated system design.

The system controls physical access based on two simultaneous conditions: a correct 4-digit PIN entered via a keypad, and a safe room temperature below 35 C. If the room is overheating - posing an equipment damage risk - entry is denied even when the PIN is correct. The OLED display shows live temperature, humidity, occupancy status, and a rolling log of the last three access attempts, providing complete real-time feedback to the user at all times.

1.2 Problem Statement

Laboratory environments at universities house expensive equipment and sensitive research materials that require controlled access. Conventional lock-and-key mechanisms do not prevent access under unsafe environmental conditions such as electrical overheating, which can cause equipment damage or fire hazards. Furthermore, manual key systems do not record who accessed a facility or when, making it impossible to investigate incidents after the fact. There is therefore a need for an intelligent, microcontroller-based access control system that can simultaneously authenticate users and monitor environmental safety, denying entry automatically when conditions are unsafe.

Section 3

Firmware / Software Development

3.1 Development Environment

All firmware was developed using the ARM Mbed online compiler at os.mbed.com, targeting the NUCLEO-F401RE. A single main.cpp source file was used containing all firmware modules. No external libraries were required - all driver implementations, including the OLED driver, DHT11 driver, and keypad scanner, were written from scratch using Mbed OS built-in APIs. The program was compiled online and the generated .bin file was downloaded and flashed to the NUCLEO board by drag-and-drop onto the board's USB mass storage drive. The board resets automatically and begins execution upon successful flashing.

3.2 Firmware Architecture and Program Structure

The firmware is structured around a single main() loop with peripheral drivers implemented as modular standalone functions. The architecture separates concerns cleanly: input scanning (keypad, button), environmental sensing (DHT11), output control (LEDs, OLED), and access logic (grant, deny, lockout, admin) are each handled by dedicated functions.

  • oled_init() / oled_cmd() / oled_dat() / oled_char() / oled_print() / oled_clear() Low-level OLED driver using I2C. Characters are rendered from a hardcoded 5-byte-per-character bitmap font for uppercase letters and digits.
  • dht11_read() Reads temperature and humidity from the DHT11 sensor over a single GPIO pin using the DHT11 one-wire protocol, including checksum validation.
  • keypad_read() Scans the 4x4 keypad by activating one row at a time and reading all columns, returning the pressed key character or 0 if no key is pressed.
  • grant_access() Toggles the occupancy flag, logs the attempt as GRANTED, lights the green LED, displays ACCESS GRANTED, and resets the PIN entry state.
  • deny_access(bool overheat) Logs the attempt as DENIED or OVERHEAT, increments the wrong attempt counter, activates the red LED, and optionally activates the yellow LED. Initiates lockout if three failures have been reached.
  • change_pin_mode() Entered when the onboard button is held. Prompts for a new 4-digit PIN via the keypad and updates the stored correct PIN.
  • show_main_screen() Renders the four OLED rows with occupancy status, PIN prompt, live temperature, and live humidity.
  • show_access_log() Displays the last three access attempt results for 3 seconds before returning to the main screen.

Environment Refresh

Lockout Handling
If locked_out is true, remaining seconds are computed from the hardware timer and displayed on the OLED. Keypad input is skipped entirely via continue.
Admin Mode
If the onboard button reads LOW (with 50 ms debounce), the system requests the current PIN before entering change_pin_mode().
Keypad Processing
A = show access log  |  # = confirm PIN entry  |  * = clear entry  |  0-9 = append digit to PIN buffer (max 4 characters).

DHT11 Read Function

main.cpp 
bool dht11_read(float &temp, float &hum) {
    uint8_t data[5] = {0};
    // Step 1: Send start signal
    dht_pin.output(); dht_pin = 0; thread_sleep_for(18);
    dht_pin = 1; wait_us(30); dht_pin.input();
    // Step 2: Wait for sensor acknowledgement
    int t = 10000;
    while (dht_pin == 1 && t--) {}
    if (t <= 0) return false;
    t = 10000; while (dht_pin == 0 && t--) {}
    if (t <= 0) return false;
    t = 10000; while (dht_pin == 1 && t--) {}
    if (t <= 0) return false;
    // Step 3: Read 40 bits
    for (int i = 0; i < 40; i++) {
        t = 10000; while (dht_pin == 0 && t--) {}
        wait_us(35);
        if (dht_pin == 1) data[i/8] |= (1 << (7 - (i%8)));
        t = 10000; while (dht_pin == 1 && t--) {}
    }
    // Step 4: Validate checksum
    uint8_t checksum = (data[0]+data[1]+data[2]+data[3]) & 0xFF;
    if (data[4] != checksum) return false;
    // Step 5: Convert data
    hum  = data[0] + data[1] * 0.1f;
    temp = data[2] + data[3] * 0.1f;
    return true;
}

void grant_access() {
    occupied = !occupied;
    log_attempt(1);
    led_green = 1; led_red = 0; led_yellow = 0;
    oled_clear();
    oled_print(0, occupied ? "LAB OCCUPIED" : "LAB VACANT");
    oled_print(1, "ACCESS GRANTED");
    oled_print(2, "WELCOME");
    thread_sleep_for(3000);
    led_green = 0; wrong_attempts = 0;
    reset_entry(); show_main_screen();
}

void deny_access(bool overheat) {
    log_attempt(overheat ? 3 : 2);
    led_red = 1; led_green = 0;
    oled_clear();
    oled_print(0, occupied ? "LAB OCCUPIED" : "LAB VACANT");
    if (overheat) {
        oled_print(1, "ACCESS DENIED");
        oled_print(2, "OVERHEAT RISK");
        led_yellow = 1;
    } else {
        wrong_attempts++;
        if (wrong_attempts >= 3) {
            oled_print(1, "LOCKED 30S");
            locked_out = true;
            lockout_timer.reset(); lockout_timer.start();
        } else {
            char msg[17] = "WRONG X LEFT";
            msg[6] = '0' + (3 - wrong_attempts);
            oled_print(1, msg);
        }
    }
    thread_sleep_for(2000);
    led_red = 0; led_yellow = 0;
    reset_entry();
    if (!locked_out) show_main_screen();
}

Project Outcomes and Accomplishments

4.1 Testing Procedure

Hardware testing was conducted by systematically exercising each functional scenario of the system on the physical NUCLEO-F401RE prototype. Each test case was executed a minimum of three times to confirm consistent and repeatable behaviour. The DHT11 threshold test was conducted by gently warming the sensor with a heat source to bring the temperature reading above 35 C and observing the system response. PIN lockout was triggered by deliberately entering three consecutive wrong PINs and observing the 30-second countdown on the OLED. Admin mode was tested by pressing and holding the onboard button, entering the correct current PIN, and setting a new PIN, then verifying that the new PIN grants access while the old PIN is rejected.

4.3 Observed System Behaviour

🖥️ OLED Display

All system states rendered correctly across four rows. Row 0 permanently displayed occupancy status. Row 1 updated contextually for PIN prompts, access results, or lockout countdown. Rows 2-3 refreshed live temperature and humidity every 3 seconds. During lockout, the countdown updated smoothly on every main loop iteration.

🌡️ DHT11 Sensor Accuracy

The DHT11 produced stable, consistent readings throughout testing. Reads were triggered every 3 seconds to respect the sensor's minimum polling interval. The checksum validation in dht11_read() reliably detected and discarded corrupted reads. On rare read failures, previously valid values were retained on the OLED without interruption.

⌨️ Keypad Response

All 16 keys were correctly identified without false triggers during repeated testing. Key release detection - polling the column line until it returns LOW - prevented multi-registration of a single press. The 100 us settle delay per row was sufficient to eliminate contact bounce artefacts.

4.5 Limitations and Possible Improvements

  • The DHT11 has +/-2 C and +/-5% RH accuracy. The 35 C threshold is set for demonstration; real deployments would typically use 40-45 C. Replacing with a DHT22 or SHT31 would improve precision.
  • The access log is stored in volatile RAM and is lost on power-off. Persisting to EEPROM or non-volatile flash would allow historical records to be retained.
  • The OLED font covers only uppercase letters and digits. Extending to lowercase and special characters would improve display readability.
  • The lockout timer uses a polling-based approach in the main loop. A hardware timer interrupt would provide a more precise and CPU-independent countdown.
  • Only a single shared PIN is supported. A multi-user system with individual PINs and unique identifiers per user would significantly increase real-world utility.
Conclusion

This project successfully demonstrates the design and implementation of a complete embedded access control system on the NUCLEO-F401RE microcontroller using the ARM Mbed development environment. The system integrates a 4x4 matrix keypad, DHT11 environmental sensor, SSD1306 OLED display, three LED indicators, and an onboard push button to deliver a functional and multi-featured access control solution.

All six project objectives were met. PIN-based authentication with lockout protection, environment-conditional access denial, real-time temperature and humidity display, occupancy tracking, access logging, and admin PIN reconfiguration were all implemented and verified in hardware. The use of Mbed OS allowed rapid development of complex multi-peripheral firmware while maintaining structured, readable, and modular code. The project demonstrates how microcontroller-based embedded systems can be applied to solve real-world engineering problems in security and environmental monitoring, and reinforces the practical skills developed throughout the EFB 2073 practical tasks completed previously.

    } else {
        wrong_attempts++;
        if(wrong_attempts>=3){
            oled_print(1,"LOCKED 30S"); locked_out=true;
            lockout_timer.reset(); lockout_timer.start();
        } else {
            char msg[17]="WRONG X LEFT";
            msg[6]='0'+(3-wrong_attempts); oled_print(1,msg);
        }
    }
    thread_sleep_for(2000);
    led_red=0;led_yellow=0; reset_entry();
    if(!locked_out) show_main_screen();
}
void change_pin_mode(){
    char new_pin[5]=""; int idx=0;
    oled_clear(); oled_print(0,"ADMIN MODE"); oled_print(1,"NEW PIN");
    while(idx<4){
        char k=keypad_read();
        if(k>='0'&&k<='9'){
            new_pin[idx++]=k;
            char dots[17]="NEW PIN ";
            for(int i=0;i<idx;i++) dots[8+i]='*';
            oled_print(1,dots);
        }
    }
    new_pin[4]='\0'; for(int i=0;i<5;i++) correct_pin[i]=new_pin[i];
    oled_clear(); oled_print(0,"ADMIN MODE"); oled_print(1,"PIN CHANGED");
    thread_sleep_for(2000); show_main_screen();
}
